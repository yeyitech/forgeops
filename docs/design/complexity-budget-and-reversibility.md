# 复杂度预算与可逆演进策略

Status: Active
Updated: 2026-03-01

## 文档定位

- 本文档用于设计与演进讨论，可能包含“目标态/候选方案”。
- 当前已落地行为以代码与 `docs/quality/verification-status.md` 为准。
- 若文档与代码冲突，请在同一 PR 同步修正文档。

## 背景

在 Agent 研发平台中，复杂度增长是必然趋势。
真正的风险不是“有复杂度”，而是“复杂度不可控、不可逆、不可验证”。

## 目标

通过“复杂度预算 + 可逆开关 + 指标验证”机制，确保平台长期演进不偏航。

## 核心原则

1. 复杂度必须有预算
- 每季度限制新增控制面复杂度总量（能力项数量/跨模块改动面）。

2. 复杂度必须可逆
- 新能力必须提供 feature flag 或明确降级路径。

3. 复杂度必须证明价值
- 2-4 周内无指标收益，默认降级或下线。

## 北极星指标

主指标（唯一）：

- `human_oversight_minutes_per_merged_pr`

推荐配套指标：

- first-pass merge rate
- reopen/rework rate
- gate escape rate
- MTTR（恢复时间）

## 复杂度分级（建议）

L1：配置级复杂度

- 参数、开关、阈值等低风险改动

L2：流程级复杂度

- workflow、gate、scheduler 的策略改动

L3：架构级复杂度

- 新增核心实体、跨层依赖、运行时契约扩展

治理要求：

- L1：可直接迭代
- L2：需变更说明与回滚路径
- L3：需 RFC、指标目标与阶段验收

## 复杂度升级阶梯（禁止跳级）

同类问题必须按以下顺序升级：

1. prompt / task 分解优化
2. skill 沉淀
3. gate / invariant 机械化
4. 平台实体化

规则：

- 未完成前一级验证，不得直接进入下一级。

## 可逆性清单（每个新能力必填）

- 开关位置（env/config/flag）
- 默认值（建议默认关闭）
- 降级动作（回退命令/配置）
- 数据兼容性（是否需要迁移）
- 观测信号（如何判断生效与失败）

## 运营节奏建议

每周：

- 复盘新增能力是否拉低主指标

每月：

- 清理低价值复杂能力（指标无收益或噪声高）

每季度：

- 评审复杂度预算执行情况
- 更新技术债清单与下季度削峰目标

## 与 ForgeOps 模型映射

- 预算执行入口：`docs/exec-plans/tech-debt-tracker.md`
- 质量观测入口：`docs/quality/domain-grades.md`、`docs/quality/verification-status.md`
- 规则承载入口：`docs/quality/golden-principles.md`

## 反模式

- 用愿景文档替代执行闭环。
- 新能力默认开启，且无降级开关。
- 指标未改善仍持续扩展能力面。

## 最小执行清单

1. 新增能力必须带可逆方案。
2. 统一用北极星指标验证价值。
3. 连续无收益能力进入下线候选。
4. 复杂问题严格走四级升级阶梯。
