# 既有项目托管与自动演进愿景（非计划）

Status: Vision (Not Planned)
Updated: 2026-02-27

> 重要声明：本文是产品愿景文档，不是执行计划，不包含排期、资源承诺或交付承诺。  
> 当前阶段仍以“新项目托管”主路径为优先，本文仅用于记录未来方向与设计边界。

## 背景

当前 ForgeOps 对新项目的脚手架与流程治理能力较完整，但对“已有存量项目”的接管仍未产品化。
真实需求不是一次性补脚手架，而是接管后能够持续识别问题、形成改进任务、并按 Harness Engineering 思路驱动可验证演进。

## 愿景目标（未来）

- 低风险接管：默认只读探测，先产出风险评估与变更计划，再由用户确认写入。
- 渐进式改造：以最小侵入方式逐步补齐治理能力，避免“全量覆盖式”改造。
- 持续问题发现：自动识别结构、质量、流程缺口，形成可追踪改进项。
- 自动任务化：将高价值问题沉淀为 issue/backlog，并保留人工审核与审计链路。
- 双循环演进：将“交付闭环”与“治理闭环”并行运行，持续提高项目健康度。

## 非目标（当前明确不做）

- 不把本文视作 active plan，不进入近期排期承诺。
- 不在当前阶段默认对已有项目执行自动写入、自动提 PR、自动改主干策略。
- 不在缺少可审计约束时启用全自动问题处置。

## 未来能力轮廓（仅愿景分层）

1. 接管探测层（Read-Only First）
- 自动扫描代码结构、工程配置、测试覆盖、文档索引与流程约束缺口。
- 输出标准化接管报告：风险等级、建议动作、影响范围、回滚建议。

2. 治理补齐层（Incremental Apply）
- 以“最小补丁”方式补齐必要治理文件与检查脚本。
- 支持 dry-run 与 apply 分离，所有写入可审计、可回放。

3. Harness 演进层（Continuous Evolution）
- 按 `docs/harness-engineering-guidelines.md` 的原则持续运行验证与改进闭环。
- 自动识别高价值问题并建议 issue 草案，保留人工确认开关。

4. 经验沉淀层（Knowledge Feedback）
- 将项目内有效改造策略沉淀为可复用技能候选。
- 与技能演化体系联动，避免“每个项目重复踩坑”。

## 设计约束（未来实现时必须满足）

- 默认安全：无显式确认不做破坏性写入与远程变更。
- 审计优先：所有自动动作必须有来源、理由、证据与结果记录。
- 可中断可恢复：任一步失败都可回退或从检查点继续。
- 人机协同：自动化用于提效，不替代关键决策的人审环节。

## 进入实施前置条件（触发立项的门槛）

- 明确“已有项目接管”最小可用范围（MVP）与成功指标。
- 完成命令拆分策略（注册、探测、补齐、应用）的用户体验评审。
- 定义安全边界与默认开关（只读、dry-run、apply、审批策略）。
- 形成可验证验收标准，并在 `docs/exec-plans/active/` 建立正式执行计划后再落地。

## 立项门槛（可量化，仍属愿景约束）

- 可控性门槛：90% 以上接管动作可在 `--dry-run` 中给出“文件级变更清单 + 风险等级 + 回滚建议”。
- 安全性门槛：默认只读模式下，误写入率为 0（以集成测试与审计日志双重验证）。
- 可恢复性门槛：关键流程失败后，95% 以上场景可在 10 分钟内通过检查点恢复继续执行。
- 问题发现门槛：在试点样本中，自动发现的高价值问题（经人工确认有效）召回率达到 70% 以上。
- 任务化质量门槛：自动生成 issue 草案被直接接受或轻量修改后接受的比例达到 60% 以上。
- 验证闭环门槛：接管后的增量治理任务中，80% 以上可自动关联验证证据（测试结果、检查报告或链接）。
- 成本门槛：单仓首次接管的人工确认耗时中位数不超过 30 分钟（不含业务改造实现时间）。
