AI DELIVERY CONTROL PLANE

把 AI 研发从“能跑”升级为“可治理交付”

ForgeOps 以 Issue 驱动 Run，用可观测状态机、可恢复会话和机械质量闸门， 将“需求到合并”变成可追踪的工程流水线。当前默认接入 Codex 运行时，并保留 Runtime Adapter 边界以支持后续扩展。

Issue 驱动执行链路

  1. Issue -> Run

    Issue-Only 模式触发 run，保证需求入口可审计。

  2. Worktree Isolation

    每个 run 独立 worktree + 分支，支持并发开发互不污染。

  3. Gates & Recovery

    Platform / Invariant / Docs Gate + step/session 恢复机制。

  4. PR & Merge

    按 workflow 配置自动合并、关闭 issue、清理 worktree。

Problem / Why

传统 AI 协作流程，常在“可运行”与“可交付”之间断裂

文档、约束、执行、验证如果不在同一控制平面里，团队会持续付出高昂的恢复与对齐成本。

过程黑箱

只看到结果，看不到 run/step/session 的完整证据链，排障成本高。

质量假绿灯

CI 通过不等于平台可运行，缺少平台 preflight/smoke 的机械约束。

知识熵增

经验无法沉淀成可复用技能，文档与流程规则随迭代持续漂移。

Solution / Features

以控制室思路重做研发流程：约束、观测、恢复、进化

ForgeOps 的核心能力基于仓库文档与实现边界，不依赖虚构功能描述。

Runtime Adapter 抽象

控制平面稳定，运行时可替换；v1 默认 `codex-exec-json`，实验支持 `codex-app-server`。

Issue-Only 强流程

run 绑定 GitHub Issue，自动创建独立 worktree/分支并推进 PR 链路。

三重质量闸门

`test` 强制平台 preflight+smoke，`implement/test/review` 强制不变量，`cleanup` 执行文档治理。

可恢复执行语义

服务重启回收 orphan step，优先续跑既有 thread，失败后回退同 step 重跑。

模式化交付路由

`quick` 优先低成本短路径，`standard` 走完整 workflow，`--local-only` 支持本地直改探索。

技能进化闭环

cleanup 产出 skill candidate，经独立 Draft PR 审核晋升到项目或 user-global 技能库。

Git Native + RL-inspired Evolution

把 Git 的可审计变更链，升级为强化学习式的人机演进闭环

ForgeOps 不训练模型权重，而是以 Git 原生交付链 + 多层上下文 + 人审晋升机制，持续优化“工程策略”而非“模型参数”。

Git Native Delivery

与 GitHub 强流程深度绑定：每次 run 都是一条可审计的变更链。

  1. Issue-Only 入口：`run create` 必须绑定 GitHub Issue。
  2. 隔离执行：自动创建 `/.forgeops/worktrees/<runId>` + `forgeops/<runId>` 分支。
  3. 基线一致性：优先基于 `origin/HEAD` 建立分支语义。
  4. 交付闭环：`implement` 自动 `commit + push + PR(draft)`，`test/review` 持续回写同一 PR。
  5. 治理收尾：`cleanup` 后可自动 merge、auto-close issue、fast-forward 同步与 worktree 归档。

对应 README「GitHub 强流程与并发 Worktree」与架构总览中的 run/worktree 机制。

Context Everywhere

上下文不是附件，而是每次 run 创建时就注入的执行前提。

任务级上下文 GitHub Issue（目标、约束、验收、偏好信号）
项目级上下文 `.forgeops/context.md` + `project.yaml` + `governance.md` + `invariants.json`
步骤级上下文 `docs/context/index.md` 注册后按 step 选择注入
角色级上下文 `agent-skills.json` + skills，按优先级解析：project-local > user-global > official

所有 run/step/session 的事件与产物可回溯到上下文，形成“输入-执行-证据-治理”闭环。

RL-inspired Loop（策略优化，不是模型训练）

  1. State 上下文状态

    Issue + context/governance/invariants + role skills

  2. Action 执行动作

    run/step 在隔离 worktree 中推进实现、测试、审查

  3. Reward 反馈信号

    done/retry/failed、gate 结果、耗时与 blocker 证据

  4. Policy Update 策略更新

    cleanup 产出 candidate，进入 trial/stable 晋升轨道

  5. Human-in-the-loop 人审裁决

    独立 Draft PR 审查决定合并/回滚，确保正确进化

该映射对齐 `skill-evolution-closed-loop` 与 `skill-promotion-pr-review-loop`：强调 Evidence First、可回滚与机械闸门，不宣称黑盒在线学习。

Skill Evolution Theater

技能不是一次配置，而是“人机共治”的可验证演进系统

基于核心概念文档，ForgeOps 的技能链路从初始化模板开始，经过项目内证据沉淀、PR 人审晋升，扩展到 user-global 复用；群体技能进化属于默认离线之上的 Opt-in 愿景层能力。

将鼠标悬浮在剧场区域，阶段会自动切换；移开后停止。

  1. 初始化技能（Baseline）

    project init 下发官方模板技能，先确保冷启动正确性。

  2. 项目技能（Project-local）

    run 过程沉淀 candidate，基于证据进入独立晋升 PR。

  3. 用户技能（User-global）

    通过 user-global 技能库跨项目复用，并保留审计链路。

  4. 群体技能（Collective，Opt-in）

    默认离线，可显式上报脱敏摘要，驱动模板级升级与回滚治理。

Phase 01 · Baseline

官方模板负责冷启动，不承诺一次最优

来源：`official-skills` + `agent-skills.json` 初始化装配。

  • Baseline First：先保证可执行，再做演进。
  • 不把模板当终点，避免“环境变了，技能不变”。
  • 角色技能优先级解析：project-local > user-global > official。
Phase 02 · Project Loop

Issue -> Delivery -> Cleanup，沉淀候选技能

来源：`issue-driven-taste-and-skill-loop` + `skill-evolution-closed-loop`。

  • Issue 步骤补齐缺口并保留 assumptions / taste signals。
  • cleanup 产出 `.forgeops/skills/candidates/*.md`。
  • Evidence First：没有 run 证据不允许晋升 stable。
Phase 03 · Human Review

正确进化依赖“自动提案 + 人审决策”

来源：`skill-promotion-pr-review-loop`（与需求流水线解耦）。

  • 晋升通过独立 worktree + Draft PR 执行，默认不自动合并。
  • 审查基线覆盖 problem / approach / evidence / scope。
  • 调度器可自动提案，但合并权仍在 reviewer。
Phase 04 · Collective (Vision)

群体技能进化是 Opt-in 愿景，不是 v1 默认承诺

来源：`skill-collective-evolution-service`（默认离线 + 显式上报）。

  • 只上报脱敏摘要，不上传源码与敏感原文。
  • collect -> aggregate -> canary -> stable，带回滚条件。
  • 目标是跨项目聚合经验，非“黑盒自动学习”。

人机协作控制点（Correct Evolution）

  1. 机器负责补齐输入缺口并生成候选，不因信息不全停摆。
  2. 机器负责输出结构化证据，升级理由必须可追溯。
  3. 人类 reviewer 对晋升 PR 做最终裁决与风险把关。
  4. 策略层保持可回滚，拒绝一次不可逆“自动升级”。
How It Works

从需求到合并，沿固定契约流动

默认步骤：architect -> issue -> implement -> test -> review -> cleanup。

  1. Run 创建：读取项目 workflow + context + governance 快照。
  2. 执行隔离：为每次 run 创建独立 worktree `/.forgeops/worktrees/<runId>`。
  3. DAG 调度：Worker 按依赖认领 pending step，可并发推进。
  4. 事件观测：API + SSE 输出 run/step/session/events/artifacts。
  5. Gate 验收：platform + invariant + docs 机械执行，失败按策略重试。
  6. 交付收尾：自动 merge/close issue/cleanup（由 workflow 配置控制）。
