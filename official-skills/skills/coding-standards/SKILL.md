# 执行准则

1. 可读性优先：命名清晰、结构稳定、避免“聪明代码”。
2. 不可变优先：默认用拷贝/返回新值，避免隐式共享状态导致回归。
3. 错误显式化：失败必须可观测（错误信息、上下文、恢复策略）。
4. 小步提交：每次改动可回滚、可验证。

## 最小规范（适用于绝大多数代码）

1. 命名
   - 变量/函数名必须表达意图，禁止 `x/q/tmp/data1` 之类无语义命名（除循环索引等）。
2. 结构
   - 入口层做校验与路由，核心逻辑下沉到纯函数/服务层，避免巨型函数。
3. 状态与副作用
   - 避免在多处写入同一对象；如果必须变更，集中到 1 个位置并加注释说明原因。
4. 错误处理
   - 捕获异常时必须补充上下文（操作对象、关键参数、可复现线索）。
   - 不吞错：禁止无脑 `catch {}` / `|| true`。
5. 并发
   - 独立 IO 优先 `Promise.all`；有顺序依赖才串行。
6. 安全边界
   - 所有外部输入（HTTP/body/query/env/file）必须在边界处校验并收敛类型。

## 输出协议（建议）

每次提交/审查结论至少包含三行：

1. `Rule`: 本次遵循/修复的关键规范点
2. `Evidence`: 对应文件与关键 diff 点
3. `Next`: 需要补的测试/验证命令
