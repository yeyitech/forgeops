# 执行准则

1. 关键路径优先：先覆盖最核心用户旅程，再扩展边缘用例。
2. 稳定性优先：禁止用 `waitForTimeout` 作为主要同步手段；用可观测条件等待。
3. 可复现证据：失败必须产出截图/trace/video（能在 CI 复现）。
4. 选择器可维护：优先 `data-testid`（或等价稳定选择器），禁止依赖脆弱的 DOM 结构。

## 最小用例集合（建议）

- happy path：登录/核心流程/提交动作
- 权限与拒绝路径：未登录、无权限、数据不存在
- 表单校验：必填/格式/边界值
- 回归点：修复过的 bug 必须固化为用例

## Flaky 处理策略（必须遵循）

1. 先定位根因：网络、动画、并发、数据污染、环境不稳定。
2. 只允许两类修复：
   - 增加稳定等待（事件/响应/locator 可见性/网络 idle）
   - 改造测试数据隔离（seed/reset、唯一数据）
3. 禁止“无限 retries”掩盖问题；如需 quarantine，必须挂 issue 并标注退出条件。

## 输出协议（建议）

每次 E2E 交付/修复后输出：

1. `Cases`: 新增/修改用例列表（文件路径）
2. `Evidence`: 运行命令 + 关键结果（PASS/FAIL，失败附 artifacts 路径）
3. `Next`: 若仍有 flaky 或未覆盖路径，列出 follow-up
